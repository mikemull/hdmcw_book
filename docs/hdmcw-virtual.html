Episode 4: Virtual Memory

A program uses memory in a few different ways. As we‚Äôve seen in our simple programs, the machine code and data we define in the program occupy memory. In more sophisticated programs, you might define variables on the fly or call subroutines which causes memory to be allocated on the stack. In other cases you might explicitly create a chunk of memory to be used for a data structure, which is allocated on the heap.

The stack and heap aren‚Äôt special types of memory‚Äì ultimately they are in the same physical memory‚Äì but they are structures tailored to the way they are being used. The stack, as the name implies, is conceptually a last-in first-out (LIFO) queue usually used for smaller chunks of memory that are transient. The heap by contrast is used for large chunks of memory, and it‚Äôs designed for that purpose. The important thing here though is that both of these sections of memory can grow (and shrink) as a program does its thing.

If my computer only ran one program at a time we could just use all of the available memory and hope that we don‚Äôt run out. That was sort of how my old PC worked. However, since you‚Äôve used a computer you probably know they run more than one thing at a time. You might be reading this on a browser, listening to JPEGMAFIA, while you ignore the document you‚Äôre supposed to be writing (i mean, i am). While you‚Äôre ignoring it, you probably don‚Äôt want the contents of that document, brilliant as they are, to take up memory. The same goes for the 40 browser tabs you have open (again, me).

We‚Äôd like to accomplish two things: use the memory in the most efficient way that allows everything to work, and also make sure that the memory we‚Äôre allocating doesn‚Äôt interfere with the memory other programs are allocating. This is what the virtual memory system, which is part operating system kernel and part hardware, does for us.

Virtual memory has two implications that we‚Äôve already hinted at. One is that the memory addresses we see in lldb or objdump are virtual. This is a little hard to grok but for now let‚Äôs just say that every location in my computer has a physical address, and what we see ain‚Äôt it. The other is that sometimes the contents of memory are temporarily swapped out to a storage device, in my case the SSD.

The command vmmap is short for virtual memory map, which is what it prints out. There‚Äôs a bunch of info here, even in my extremely elided version. Let‚Äôs take a look at the highlights.

Our ~15 line assembly language program has a ‚Äúphysical footprint‚Äù of nearly a megabyte, which is weird. The part that our code uses is covered by the first line of the non-writable regions (__TEXT 102810000-102814000) Most of the rest here is either dynamic libraries (the .dylib things) or things related to stack and heap memory (memory on the heap is usually allocated with the C library function malloc(), hence all the MALLOC_ stuff). Note also near the top it says VM page size: 16384 bytes. The idea of a page is essential to virtual memory. What this means is that the virtual memory space is divided up into 16 K chunks (the page size is always a power of 2). When our program attempts to access something at a particular virtual memory address two things happen. The virtual address is translated to a physical address, and then there‚Äôs a check to see if the page in which that instruction or data lives is already resident in physical memory. That happens in the hardware because it needs to be as fast as possible. However, if the page is not resident, the hardware just returns a page fault which indicates to the operating system that it needs to retrieve the data from the SSD.

Another couple of things to notice here: in the ‚ÄúWritable‚Äù section there‚Äôs a big gap between the MALLOC_SMALL section and the Stack section (like, a billion). That‚Äôs because usually the virtual memory space is set up so that the stack grows downward and the heap grows upward, address-wise. Also notice there is the SHRMOD (share mode) column, and for many things especially the ‚Äúdylib‚Äù entries, the share mode is ‚ÄúSM=COW‚Äù. No bovine connection, COW means ‚Äúcopy on write‚Äù. This is a common optimization that basically means you share the memory until a process wants to modify something then you make a copy of it specifically for that process. Since a process is unlikely to modify the code of a system library it‚Äôs fairly safe to assume these will remain COWs.

There‚Äôs a ton more to virtual memory, but i think we‚Äôve got enough to go on. We haven‚Äôt really discussed the mechanics of how this virtually memory map gets built, but this gives us an idea of how code and data gets into memory so that the program can start execution.
